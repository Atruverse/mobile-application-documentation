Logic Structure

This app is structured with a feature based approach. Each feature is a library which consists of three parts

  • UI
  • Controllers
  • Repositories

UI

This part includes pages in which many widgets are nested.

Controllers

All the functionalities of a page are handled inside a separate controller that is attached to the page. Each controller has to be bound before using

Controller binding

Is the process of letting GetX know that a controller exists. If not done, pages cannot make use of their controllers since GetX doesn't know of their controller's existence. In this project, controller binding takes place inside the core folder of any library separately. Here is a sample code of binding a controller.

Get.lazyPut<TodoRepository>(
   () => TodoRepositoryImp(),
   fenix: true,
);

fenix: true means that this controller or repository is able to be reconstructed in case of disposal

Repositories

Controllers make use of repositories for API calls and database tasks. Each repository consists of many methods which are customized to perform a specific task. Binding repositories is also mandatory and the syntax is the same as controller binding

Folder Structure

This app consists of three main folders.

  • core
  • features

Core Folder

This folder consists of all the reusable classes and data that will be used by features. For example app routes or server routes are defined in this folder since they will be used in almost every page or controller.

Features Folder

As its name implies, this folder consists of the features that make up the app. Inside this folder, each feature has its own separate folder which consists of some subfolders that provide the feature with its desired functionality. Every feature has a library.dart folder in which all the files that other features need from this feature are exported. Moreover, you can find a complete description of what the feature does and how it works inside the library.dart file located inside each feature's folder.

Development Protocols

There are some rules that has been followed all across this source code and I recommend following them in order to keep the code cohesive. These rules are listed accordingly.

  • Don't use any other state management approach other than GetX, this will make debugging much harder since two different logics have been used in different parts so try to narrow down your mindset to GetX and try to solve each issue using GetX.
  • Only use GetX for routing
